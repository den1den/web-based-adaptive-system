Creating a Login or Registration Page

By looking at the source of the tutorial login page and the registration page you can get a good idea what these pages must contain. We briefly summarize the attributes and fields that can be present in the login and registration forms:

  • action: Every HTML form must have an action attribute. It is the name of the servlet to call. It must be "/aha/Register" on the registration page and "/aha/Login" on the login page. For anonymous login and registration the action is "/aha/Anonymous".
  • name: This is the user's name. It can be shown anywhere in pages but is typically shown in headers or footers, and it can be used in rules. So can all other fields that appear on the login and registration forms. The name field has no special function.
  • email: This is the user's email address.
  • login: This is the user's login id. It must be unique in the system. It is used when logging in again.
  • password: This is the user's password. It should be kept secret and be different from passwords used with other systems. AHA! currently does not store passwords in a very secure way.
  • start: This is the name of the first concept to be shown after registering or logging in. This is typically a hidden field. It is mandatory: AHA! does not know which page to show without this field.
  • course: This is the name of the AHA! application. (The name "course" was chosen for historical reasons.) The application name is used in conjunction with the concept structure. This is typically a hidden field. It is mandatory. AHA! will not work when this field is missing.
  • directory: This is the name of the directory on the server containing the application. In principle this can be different from the application's name, but not all tools are guaranteed to recognize applications where the name is different from the directory.
  • title: The title that can be used to refer to the application. The title can have several words whereas "course" can be only one word.
  • background: This is the name of an optional background image to be used on the application's pages (in XHTML+AHA format).
  • Other fields may be used as well. They result in attributes of the "personal" concept in the user model which can be used in the rules.
  • The submit button must have the correct name to be recognized by AHA!. It is "enter_button" on the registration form and
  • For an anonymous login there is a separate form with "/aha/Anonymous" as its action. In the form the submit button must have a "name" field with value "session_button" on the registration page and "resume_button" on the login page. Other than that the form may contain the same fields as for non-anonymous logins, although there are typically no fields to be filled out. The only fields that cannot be present (or will be ignored) are "login" and "password".
